Coherent Code Skills · 01

Vibes don't ship. Workflow does.

Claude Code is a brilliant junior who skips steps. It edits before reading. It claims success before testing. It writes code before there's a plan. Install cd-skills and the junior turns into a senior — twelve skills that gate every phase of the loop.

11 slides · ~5 min read · ~3 min install Claude Code + git + (optional) gh macOS · Windows · Linux
Outcome

Three artifacts. One coherent loop.

By the end Claude moves through your work like an engineer who's been there before — with hard gates between phases instead of vibes.

A plugin

cd namespace, 12 skills under cd:<name>. Loads on demand. A 500-byte index injects on session start so Claude knows the loop exists without paying the cache tax.

A canonical loop

context → brainstorm → plan → execute → review → ship. Each skill ends with a forced handoff. Claude can't drift past a gate without your approval.

A safety net

Verification baked into every step. cd:debug bans guess-and-check fixes. cd:ship refuses to push with red tests. The agent can't lie to itself.

The mental model
Think of cd-skills as a checklist your tablesaw refuses to start without. Annoying for one cut. Lifesaving for ten thousand.
Before you start

What you need on hand.

  1. Claude Code, any version with plugin support. If you can run /plugin marketplace add in a session, you're fine. CLI, desktop, or IDE extension all work the same.
  2. Git, anywhere. Most cd skills (plan, execute, ship, worktree) hand Claude commands like git status and git log. A repo-less workspace still works for brainstorm and memory, but you'll get the most out of cd-skills inside a git project.
  3. (Optional) GitHub CLI gh. Needed only if you want cd:ship to open PRs for you. Without it, the skill falls back to "push the branch" or "merge locally."
  4. ~3 minutes. Install: 30 seconds. First /clear to fire the session-start hook: 2 seconds. Trying cd:context on a real repo: ~2 minutes.
No new accounts, no new tools
cd-skills is just markdown + a tiny shell hook. No backend, no telemetry, no API key. The plugin is MIT-licensed at github.com/Coherence-Daddy/cd-skills.
Step 01

Install cd-skills.

Three slash commands inside Claude Code. No clone, no PATH, no daemon.

Inside any Claude Code session
/plugin marketplace add https://github.com/Coherence-Daddy/cd-skills.git
/plugin install cd@cd-skills
/reload-plugins

Then /clear (or open a new session) so the SessionStart hook fires. That's the hook that injects the <cd-skills> index — the model now knows the workflow exists.

Verify
Type /cd: and pause. Autocomplete should list 12 skills: brainstorm, context, debug, execute, memory, new-skill, parallel, plan, review, ship, tdd, worktree.
If your terminal swallowed the URL
Some shells choke on the .git suffix or the https:// prefix when typing inside a slash command. The literal string the harness wants is https://github.com/Coherence-Daddy/cd-skills.git — paste, don't retype.
Step 02

Meet the 12 skills.

Six are workflow phases. Five are cross-cutting. One is meta. Each lives in its own SKILL.md with a hard gate, a checklist, and a forced handoff.

cd-skills/skills/ ├── context ← load enough before acting · know / assume / verify gate ├── brainstorm ← idea → written spec · scope decomposition ├── plan ← spec → checkbox plan · approval gate ├── execute ← walk plan step by step · per-step verify ├── tdd ← test-first mode of execute ├── parallel ← independent steps → subagent fan-out ├── review ← code review · mode=request | mode=receive ├── ship ← verify → commit → integrate · refuses red tests ├── debug ← root-cause-first · bans guess-and-check ├── worktree ← isolated tree for parallel or risky work ├── memory ← recall / save / audit auto-memory · 3 modes └── new-skill ← author your own in cd format
The pattern every skill follows
Hard gate (a code-fenced block of constraints) → Checklist (numbered, mappable to tasks) → Process (the actual work) → Anti-patterns (failure modes named explicitly) → Handoff (what skill or action follows). Read one and you've read them all.
Naming convention
Phase skills are verbs (brainstorm, plan, execute). Cross-cutting skills are short context names (tdd, debug, worktree). Anything you'd say out loud as "let's __ this," that's the slug.
Step 03

The canonical loop.

Six phases, gated end to end. Each ends by handing off to the next — Claude can't drift past a gate without your approval.

cd:context → load memory + git + key files; produce know/assume/verify block ↓ cd:brainstorm → one-question-at-a-time dialogue; spec saved + committed ↓ cd:plan → spec becomes checkbox plan; you approve, plan committed ↓ cd:execute → one step at a time, with quoted verify output per step ↓ cd:review → self-review then dispatch the code-reviewer agent ↓ cd:ship → full verify, then YOU pick: merge / PR / push / stop
Law 01

Every phase ends with a written artifact.

Brainstorm produces docs/specs/<date>-<slug>.md. Plan produces docs/plans/<date>-<slug>.md. Execute ticks checkboxes after verify passes. The artifacts let any future session pick up cold.

Law 02

Every gate refuses to pass through.

cd:execute won't tick a step without quoted verify output. cd:ship won't push with red tests. cd:debug bans guess-and-check fixes. Constraints over guidelines, always.

Analogy: pre-flight
A pilot doesn't take off because they "feel ready." They run a written checklist that another pilot wrote. cd-skills is that checklist for shipping software with an agent.
Step 04

Five skills that cut across the loop.

These don't replace a phase — they sharpen one. Invoke them when the situation calls.

  1. cd:tdd — test-first execution. Mode of cd:execute. Write the failing test, watch it fail (quote the failure), write the minimum code, watch it pass, commit. Skip for refactors and config — TDD is for new behavior in a tested codebase.
  2. cd:parallel — fan out independent steps. If your plan has 2+ steps with no shared state, dispatch them as parallel subagents. The skill verifies independence first (no two agents touch the same file) and merges results. Like a relay team where every runner has a separate lane.
  3. cd:debug — root-cause-first. Bans the agent's seven favorite anti-fixes: empty catch, null-coalesce silencing, version pinning, cache clears, sleeps in race conditions, gagging the test, restart-and-pray. Forces a written hypothesis chain before any change.
  4. cd:worktree — physical isolation. For when you don't want to disturb the current tree — bug fix on a side branch, parallel agents that each commit, risky experiment. git worktree add ../<slug> -b feature/<slug> with a setup checklist and a cleanup gate.
  5. cd:memory — recall, save, audit. Wraps the harness's auto-memory store with three modes. recall returns a cited summary on a topic. save enforces the format (type, why, how-to-apply) so memories don't rot. audit reviews the whole store and proposes prunes — never deletes silently.
When to invoke vs. let it auto-fire
Skills with strong triggers (cd:debug on a test failure, cd:context at session start) the model picks up automatically from the SessionStart index. The cross-cutting ones you'll want to call by name when you know the situation: /cd:parallel, /cd:worktree, etc.
Step 05

Run the loop, end to end.

Open a real repo. Pick any small task — a feature, a bug fix, a refactor. Then drive Claude through the full loop with five short messages.

In Claude Code · paste these one at a time
1. /cd:context

2. I want to add <your task>.

3. /cd:plan

4. /cd:execute

5. /cd:ship

Message 1 forces a know/assume/verify block before anything happens. Message 2 trips cd:brainstorm automatically (its description fires on "I want to add"). Messages 3-5 chain the rest. After each, Claude reports what changed and waits.

What you'll see

  • A spec file in docs/specs/
  • A plan file in docs/plans/ with checkboxes
  • One commit per plan step
  • A code-review report categorized BLOCKER / SUGGEST / NIT
  • An integration menu (merge / PR / push / stop) that won't pick for you

Where it'll feel different

  • Claude asks one question at a time, not five
  • It pauses after every plan step for verify
  • It refuses to push if anything's red
  • It cites file:line in review findings, not "around here"
First-time hint
Pick a task you'd normally do in 20 minutes. The loop adds maybe 5 minutes of overhead — and saves the hour you'd spend untangling a half-baked diff. That ratio improves as the task gets larger.

Hand off the work

Drop this prompt into Claude. Watch the kit land itself.

If you'd rather have Claude do the install and drive the first loop pass — paste this and it takes the wheel.

Copy-paste prompt
You are my Coherent Code Skills · 01 install operator. Goal: install the
cd-skills plugin in my Claude Code, verify all 12 skills loaded, and walk me
through one full pass of the canonical loop on a small task in my current
repo. Do 95% of the work yourself.

Steps you own:
1. Run /plugin marketplace add https://github.com/Coherence-Daddy/cd-skills.git
2. Run /plugin install cd@cd-skills
3. Run /reload-plugins, then ask me to /clear so the SessionStart hook fires.
4. After /clear, confirm the <cd-skills> index appears and that the 12 skills
   (context, brainstorm, plan, execute, tdd, parallel, review, ship, debug,
   worktree, memory, new-skill) are listed.
5. Run cd:context on the current repo. Produce the know/assume/verify block.
6. Ask me what small task I want to use as the dogfood task (one sentence).
7. Run cd:brainstorm on that task, asking ONE question at a time. Save the
   spec to docs/specs/<date>-<slug>.md and commit.
8. Run cd:plan. Save to docs/plans/<date>-<slug>.md and commit.
9. Run cd:execute, one step at a time, with quoted verify output per step.
10. Run cd:review (mode=request). Address blockers, defer suggestions.
11. Run cd:ship. Show me the integration menu — DO NOT pick for me.

Rules:
- Render the full to-do list as a markdown checklist in your first reply.
  Update it after every completed step.
- One skill at a time. Hand off explicitly between phases.
- Never push, merge, or PR without my explicit pick from the cd:ship menu.
- Never edit code outside the active plan step. If you need to touch a
  file the plan doesn't list, stop and update the plan first.
- Quote verify output verbatim. "Tests pass" without quoted evidence is
  not a completion.

Start by asking my project repo path and confirming Claude Code has plugin
support enabled, then begin step 1.
